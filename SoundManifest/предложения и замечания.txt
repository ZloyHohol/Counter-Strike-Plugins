Из‑за этого каждое убийство в раунде воспринималось как FirstBlood, потому что счётчик totalKills мы не вели, а g_bFirstBloodDone сбрасывался только на round_start.

Как это было реализовано в старом QuakeSounds
В quakesounds_css.sp логика выглядела так:

В начале раунда:

sourcepawn
totalKills = 0;
При каждом убийстве:

sourcepawn
totalKills++;
if (totalKills == 1 && settingConfig[FIRSTBLOOD][NOT_BASED_ON_KILLS]) {
    soundId = FIRSTBLOOD;
}
То есть первая кровь определялась не флагом, а фактом: «это первое убийство в раунде».

Дальше шли проверки на killstreak, combo, headshot и т.д. — строго по приоритету.

Что нужно вернуть в SoundManifest
Добавить глобальную переменную:

sourcepawn
int g_iTotalKills = 0;
Сбрасывать её в OnRoundStart:

sourcepawn
g_iTotalKills = 0;
В OnPlayerDeath инкрементировать:

sourcepawn
g_iTotalKills++;
Проверку FirstBlood заменить:

sourcepawn
if (g_iTotalKills == 1) {
    PlayEvent("FirstBlood", iAttacker, iVictim);
}
Убрать g_bFirstBloodDone, он больше не нужен.

Приоритет событий
Старый QuakeSounds делал так:

Suicide / TeamKill → сразу.

FirstBlood → если totalKills == 1.

Killstreak (consecutiveKills).

Combo (lastKillCount).

Headshot (по счётчику или флагу).

Knife/Grenade.

Это гарантировало, что одно событие = один звук, и FirstBlood никогда не повторялся.
